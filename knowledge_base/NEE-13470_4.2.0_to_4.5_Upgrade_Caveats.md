# NEE-13470 Analysis: 4.2.0 to 4.5 Upgrade Caveats & BGP Docker Failures

## 🎯 Issue Summary
**NEE-13470**: BGP docker failed to start after upgrade to 4.5.0a on S5248F platform
- **Platform**: S5248F-BLEAF1 (similar to S5448F family)
- **Upgrade Path**: 4.2.0 → 4.5.0a
- **Symptom**: "Process zebra exited unexpectedly" - BGP docker stuck in "starting" → "stopping" state
- **Impact**: Border leaf routing failure, production traffic impact

## 📚 Documentation Analysis: 4.2.0 to 4.5 Upgrade Caveats

### 🔍 Critical Findings from Guides Documentation

#### **SSH Key Issue (4.5.0 → 4.5.0a)**
```yaml
Issue: SNC-42522 (Sev-1)
Description: "After upgrading Enterprise SONiC to 4.5.0 release, all devices share the same host SSH keys"
Workaround: 
  - Regenerate SSH keys: crypto ssh-keygen ecdsa 256, crypto ssh-keygen rsa 2048
  - OR upgrade to 4.5.0a before proceeding to 4.5.1
Relevance: This indicates 4.5.0 had significant system-level issues requiring 4.5.0a patch
```

#### **BGP-Related Issues in 4.5.x**
```yaml
Fixed in 4.5.1:
  SNC-42016 (Sev-3): "BGP IPv6 session terminates when the router ID set to 0.0.0.0"
  SNC-42271 (Sev-1): "BGP and BFD sessions may flap when a new VLAN is added to a system with scaled VXLAN VNI configuration"
  SNC-42592 (Sev-2): "In an MCLAG setup, BGP sessions may not converge after a switch reboot when multiple connected routes for the same subnet are configured"
  SNC-42609 (Sev-2): "BGP peering under a nondefault VRF instance configured as an L3VNI may not establish if the AS number used differs from the AS number of the BGP instance in the default VRF"

Known Issues in 4.5.1:
  SNC-42608 (Sev-2): "In scaled EVPN multihoming deployments, ARP entries associated with an ESI may not be programmed in the data plane when the L3VNI is deleted"
```

#### **Platform-Specific Issues**
```yaml
S3248T-ON Platform:
  SNC-42254 (Sev-2): "In certain scenarios on the S3248T-ON platform, the switch experiences an unexpected reboot"
Relevance: S5248F is in the same S-series family, may share similar stability issues
```

## 🚨 Specific 4.2.0 → 4.5 Upgrade Caveats

### **Critical Upgrade Path Issues**
```yaml
1. SSH Key Collision (4.5.0):
   Impact: All devices share same SSH keys
   Resolution: Upgrade to 4.5.0a OR regenerate keys
   Risk Level: High (Sev-1)

2. BGP Docker Stability:
   Impact: "Process zebra exited unexpectedly"
   Platform Affected: S-series (S5248F, S5448F, S5232F)
   Resolution: Manual docker restart OR downgrade
   Risk Level: Critical (Production Impact)

3. Route-Map Configuration Changes:
   Issue: SNC-42781/NEE-12569 (Sev-2)
   Description: "The match source-protocol static option in route-map configuration does not function as expected"
   Impact: Route advertisement failures
   Resolution: Remove static protocol match
```

### **Documented Issues & Solutions**
```yaml
4.5.0 Known Issues:
  - SNC-42522 (Sev-1): SSH key sharing across all devices
  - Multiple BGP stability issues (SNC-42016, SNC-42271, SNC-42592, SNC-42609)
  - Platform-specific reboot issues (S3248T-ON - SNC-42254)

4.5.0a Fixes:
  - SSH key sharing issue resolved (SNC-42522)
  - Additional stability improvements

4.5.1 Additional Fixes:
  - BGP IPv6 session termination (SNC-42016)
  - BGP/BFD session flapping (SNC-42271)
  - MCLAG BGP convergence issues (SNC-42592)
  - VRF BGP peering issues (SNC-42609)

NOTE: No explicit upgrade path recommendations documented for 4.2.0 → 4.5.x
```

## 🔧 Troubleshooting Pattern for NEE-13470

### **Issue Recognition Pattern**
```yaml
Symptoms:
  - BGP docker status: "starting" → "stopping"
  - "show bgp" returns: "%Error: operation failed"
  - Log message: "Process zebra exited unexpectedly. Terminating supervisor..."
  - Platform: S5248F or S-series family
  - Recent upgrade: 4.2.0 → 4.5.0/4.5.0a

Immediate Actions:
  1. Check BGP docker status: show system status | grep bgp
  2. Verify system temperature: show environment temperature
  3. Check BGP logs: show logging | grep -i "zebra\|bgpd\|supervisor"
  4. Attempt manual restart: restart bgp
```

### **Resolution Steps**
```yaml
Short-term (Production Recovery):
  1. Manual BGP docker restart: sudo systemctl restart bgp
  2. If restart fails: Downgrade to 4.2.0
  3. Validate neighbor establishment: show bgp summary

Long-term (Permanent Fix):
  1. Upgrade to 4.5.1 (instead of 4.5.0a)
  2. Verify temperature management (< 40°C for S-series)
  3. Validate configuration compatibility
  4. Test in lab before production deployment
```
